找工作的准备工作

发表于2023-10-02|更新于2023-10-04|job Prepare
|字数总计:2.6k|阅读时长:8分钟|阅读量:

js 基本数据类型

String、Number、Boolean、Undefined、Null、Object、Symbol（ES6新增）
  • isNaN(Not a Number)：用于判断参数是否是非数字值的内置函数

url 解析过程

  • 浏览器输入 url 地址，首先要在客户端上进行 url 解析
  • 浏览器会先查看自身的缓存，如果浏览器缓存中有对应的解析记录，直接返回结果如果浏览器没有缓存，电脑会查看本地操作系统的中的缓存，如果有记录，直接返回结果（host 文件）如果本地没有缓存域名的 ip 地址，就需要通过迭代或者递归的方式向根域名服务器、顶级域名服务器、权威域名服务器发起查询请求，直接返回一个 IP 地址给浏览器
  • 然后根据 IP 地址建立 TCP 连接（三次握手）
  • 客户端发送 HTTP 请求，服务器返回报文，关闭 TCP 连接（四次挥手）
  • 浏览器拿到相应内容后，进行解析：解析 html、css 分别建立 DOM Tree 和 CSSOM Tree，然后合并成 Render Tree，然后布局 Render Tree，绘制 Render Tree，绘制页面像素信息。将各层的信息发送给 GPU，GPU 将各层合成，显示在屏幕上

TCP 三次握手与四次挥手

三次握手

三次握手 客户端 发送信息 服务端
第一次握手 1、客户端即不知道自己发送与接收的能力是否正常 2、客户端也不知道服务器发送与接收的能力是否正常 SYN=1，seq=x 服务端收到客户端发送来的消息 1、服务器知道客户端发送正常，不知道客户端接收能力是否正常，2、服务器知道自己接收能力正常，不知道自己发送能力是否正常
第二次握手 客户端收到服务端发来的消息 1、客户端知道自己发送、接收能力正常 2、客户端知道服务端发送、接收正常 客户端全部知道 ACK=1，ack=x+1，SYN=1，seq=y 服务器发送消息给客户端
第三次握手 客户端发送给服务器 seq=x+1，ACK=1，ack=y+1 服务器收到客户端发来的消息 1、服务器知道客户端发送，接收正常 2、服务器知道自己发送、接收正常 服务器全部知道 
从上述分析可知，握手两次达不到双方都知道自己、对方的接收、发送能力都正常的结论。由此我们知道三次握手的目的是让客户端、服务端都知道自己和对方的发送、接收能力是正常的，所以三次握手不能少，多了浪费网络资源

四次挥手

四次挥手 客户端 发送信息 服务端
第一次挥手 客户端发送请求 FIN-WAIT-1 状态 1 FIN=1，seq=u
第二次挥手 ACK=1，ack=u+1，seq=v 服务端收到请求 服务端发送请求 CLOSE-WAIt 状态 2
第三次挥手 客户端收到服务器确认结果后进入 FIN-WAIT-2 客户端发送 状态 3 FIN=1，ACK=1，ack=u+1，seq=w 服务端发送消息 LAST-ACK 状态 4
第四次挥手 客户端收到回复 客户端发送 TIME-WAIT 状态 客户端经过两个最长报文寿命后，客户端 CLOSE5 ACK=1，ack=w+1， seq=u+1 服务端确认 服务端立刻进入 CLOSE 状态 6 
**四次挥手需要考虑服务端是否有正在发送的数据** 所以客户端发起结束请求，服务端不能立刻响应结束。简单理解：客户端发起结束请求 -> 服务端收到请求，同时告诉客户端自己收到请求 -> 服务端确认没有正在发送的数据，告诉客户端 -> 客户端收到服务端的确认后结束，服务端结束

正是因为 TCP 协议有三次握手与四次挥手，所以 TCP 是安全连接，区别于 UDP 协议

HTTP 与 TCP 的关系

  • TCP 是传输层协议，而 HTTP 是应用层协议
  • HTTP 基于 TCP 连接基础上，所以三次握手有人说是 HTTP 的，有人说是 TCP 的，准确说是 TCP
  • TCP 就是单纯建立连接，不涉及任何需要请求的实际数据，简单的传输
  • HTTP 是用来收发数据的

cookie、session、localStorage

cookie

由于HTTP 协议是无状态的，所以服务器不知道用户上一次做了什么，影响交互式 web 的实现，所以出现了 cookie，cookie 的作用就是在客户端存储用户访问网站的一些信息，典型应用：

  • 记住密码，下次自动登录
  • 购物车功能
  • 记录用户浏览数据，进行商品（广告）推荐
  • 个性化设置（主题等）

cookie 的缺点

  • cookie 会被附加在每个 HTTP 请求中，所以无形中增加了流量
  • 由于在 HTTP 请求中中的 cookie 是明文传递的，所以安全性成问题（除非使用 HTTPS）
  • cookie 的大小只有 4kb 左右，对于复杂存储需求来说不够用
  • 会话 cookie：若不设置过期时间，cookie 的生命周期为浏览器会话期间，关闭浏览器窗口，cookie 就会消失
  • 持久 cookie：设置了过期时间，cookie 保存到硬盘上，关闭在打开浏览器，这些 cookie 仍然有效直到超过设定的过期时间，可以在浏览器的不同进程之间共享
  • cookie 具有不可跨域名性：就是说浏览器访问百度不会带上谷歌的 cookie
如果用户禁用 cookie，则要使用 URL 重写，可以通过 response.encodeURL(url)进行实现，将 sessionID 拼接到访问地址后

session

  • session 是一种服务器机制，用来存储特定用户会话的信息
  • 第一次访问服务器的时候，会创建一个 session ID，之后的访问服务器首先检查用户发来额请求里是否包含了一个 session ID，如果包含，就在服务器的内存中查找出来，此 session ID 唯一、不重复的字符串，响应后返回到客户端保存，存储在 cookie 中

典型的应用

  • 判断用户是否登录
  • 购物车功能
  • 将某些数据放入 session 中，供同一用户的不同页面使用
  • 防止用户非法登录
cookie与session（会话跟踪技术）、H4的存储机制；localStorage、sessionStorage是Web Storage，H5的本地存储机制。

session 的缺点：

  • session 保存的东西越多，越占用服务器内存，对于在线人数多的网站，服务器内存压力大
  • 依赖于 cookie，如果禁用 cookie，则要使用 URL 重写，不安全
  • 创建 session 变量有很大的随意性，可随时调用，不需要开发者做精确的处理，所以过度使用 session 变量将会导致代码不可读而且不好维护

cookie 与 session 的对比

  • cookie 只能保存字符串类型，以文本的方式；session 通过类似 Hashtable 的数据结构来保存，能支持任何类型的对象（session 中可以包含多个对象）
  • cookie 保存的大小只有 4kb，session 没有大小限制
  • cookie 存在的攻击：cookie 欺骗、cookie 截获；session 的安全性大于 cookie
  • cookie支持跨域，将 domain 属性设置为“.bdf.com”，则以“.bdf.com”为后缀的一切域名能够访问 cookie，session 不支持跨域名访问，仅在所在域名内有效
  • session 不区分路径，在访问同一个网站期间，所有的 session 在任何一个地方都可以访问到，而 cookie 中如果设置了路径参数，那么同一个网站中不同路径下的 cookie 互相是访问不到的

session 安全性高的原因：

  • session ID 存储在 cookie 中，若要攻破 session，首先要攻破 cookie
  • session ID 是要有人登录，或者启动 session_start 才会有，所以攻破 cookie 也不一定得到 session ID
  • 第二次启动 session_start 后，前一次的 session ID 就失效了，session 过期后，session ID 也随之失效
  • session ID 是加密的

localStorage 和 sessionStorage 的对比

生命周期

  • localStorage：生命周期是永久的，关闭页面或者浏览器之后 localStorage 中的数据也不会消失，除非主动删除数据，否则数据永远也不会消失
  • sessionStorage：生命周期仅在当前会话有效，其在同源的窗口中始终存在的数据。只要这个浏览器窗口没有关闭，即使刷新页面或者进入另一个页面，数据依然存在。但是关闭了浏览器窗口后就会被销毁，同时独立的打开同一个窗口同一个页面，sessionStorage 也是不一样的

作用域

  • sessionStorage 在不同的浏览器窗口中不共享，不同页面或标签页间无法共享 sessionStorage 的信息
  • localStorage 在所有同源窗口中都是共享的，相同浏览器的不同页面间可以共享相同的 localStorage（页面属于相同域名和端口）

localStorage 和 sessionStorage 的存储数据大小一般都是 5MB
使用方式：window.localStorage 和 window.sessionStorage
存储类型：都只能存储字符串类型，对于复杂的对象可以使用 ECMAScript 提供的 JSON 对象的 stringify 和 parse 来处理
localStorage 常用于长期登录，适合长期保存在本地的数据；sessionStorage 敏感性账号一次性登录

对比 cookie，Web Storage 的优点

  • 存储空间大：cookie 4kb，Web Storage 5MB
  • 节省网络流量：Web Storage 不会传送到服务器，存储在本地的数据可以直接获取，也不会像 cookie 一样每次请求都会传送到服务器，减少了客户端和服务端的交互，节省了网络流量
  • 敏感访问：对于那种只需在用户浏览一组页面期间保存而关闭浏览器后就可以丢弃的数据，sessionStorage 会非常方便
  • 快速显示
  • 安全性
  • 操作更方便：Web Storage 的 API 更加简单
